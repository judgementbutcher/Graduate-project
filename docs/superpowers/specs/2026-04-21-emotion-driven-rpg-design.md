# 基于情感计算的生成式叙事 2D RPG 毕业设计方案

## 1. 项目定位

本项目定位为一个小体量、强交互、强叙事的 2D RPG 毕业设计原型。系统以 `Godot` 作为前端表现层，以 `FastAPI` 作为后端服务层，以 `SQLite` 作为本地数据持久化方案。项目目标不是开发完整商业 RPG，而是在可控工期内实现一个可运行、可演示、可写论文、可答辩的原型系统。

本项目的核心卖点是：玩家输入会通过情感感知、记忆检索和任务约束融合决策，影响 NPC 的行为选择和对话反馈。系统重点不在战斗、数值成长或大地图探索，而在 NPC 智能交互与生成式叙事机制。

## 2. 范围与目标

### 2.1 总体目标

项目需要同时满足两个目标：

- 系统目标：实现一个可运行的 2D RPG 原型，支持地图探索、NPC 对话、任务推进、状态变化和存档读档。
- 论文目标：实现一套可解释、可验证的 NPC 叙事决策算法，并通过实验案例说明算法有效性。

### 2.2 范围边界

项目采用 MVP 控制策略，最终范围固定为：

- 2 个场景：村庄、森林入口或废塔外部
- 3 个关键 NPC：村长、守卫、商人
- 1 条主线任务
- 2 条支线任务
- 1 套完整的对话、任务、关系、存档闭环

不纳入本次实现范围的内容包括：

- 战斗系统
- 背包合成与复杂成长系统
- 大地图或开放世界结构
- 大量 NPC 和复杂任务树
- 联网、多玩家、账号系统
- 以大模型为核心的自由生成式对话系统

## 3. 技术路线

### 3.1 技术选型

- 前端：Godot 4.x
- 后端：Python 3.11+、FastAPI
- 数据库：SQLite
- ORM：SQLAlchemy
- 数据校验：Pydantic
- 服务运行：Uvicorn

### 3.2 技术策略

项目优先发挥作者在 `Godot` 前端上的优势，前端负责视觉演示和交互反馈；后端只保留对答辩和论文真正重要的状态管理与决策逻辑。文本生成采用“模板为主、LLM 润色为辅”的策略。LLM 仅作为增强项，不是系统核心。

美术资源策略为“优先现成素材整合，少量自定义修改”，以降低内容制作成本，把时间留给系统闭环和核心算法。

## 4. 系统架构

系统采用轻量前后端分离架构：

- `Godot` 前端负责地图场景、玩家移动、NPC 交互、对话 UI、任务 UI、存档界面和接口调用。
- `FastAPI` 后端负责情绪分析、记忆检索、任务约束判断、候选行为评分、任务推进和状态存储。
- `SQLite` 持久化玩家状态、NPC 状态、任务进度、记忆信息和对话日志。

### 4.1 前端职责

前端仅承担以下职责：

- 地图场景切换
- 玩家移动与触发交互
- 对话框、任务面板、存档界面展示
- 向后端发送交互请求
- 接收后端结果并刷新 UI、任务状态和 NPC 表现

### 4.2 后端职责

后端仅承担以下核心模块：

- `dialogue_api`：接收交互请求并返回 NPC 回复结果
- `emotion_service`：对玩家输入做简化情绪识别
- `memory_service`：检索 NPC 历史记忆
- `decision_service`：融合情绪、记忆、关系、任务状态做行为选择
- `quest_service`：维护任务状态和推进逻辑
- `save_service`：处理存档、读档和日志记录

### 4.3 架构边界

系统必须遵守一条边界原则：复杂状态与决策逻辑全部放在后端，`Godot` 不直接修改数据库，也不承载核心算法。前端只通过接口与后端通信。这样可以保证系统结构清晰、职责明确，并便于在论文中解释模块分工和数据流。

## 5. 核心玩法循环与数据流

本项目的核心体验是“对话 + 任务推进”，而非战斗。玩家的核心循环如下：

1. 在场景中探索并接触 NPC。
2. 触发交互界面，选择预设选项或输入文本。
3. 前端将 `player_id`、`npc_id`、`scene_id`、`input_text`、`quest_state` 发送至后端。
4. 后端完成情绪识别、记忆检索、任务约束判断和行为评分。
5. 后端生成结构化交互结果，包括 NPC 行为、回复文本、关系变化和任务变化。
6. 前端展示 NPC 回复，并更新任务面板、关系状态和场景反馈。
7. 玩家继续对话、探索或推进任务。

### 5.1 交互模式

为保证稳定性和可控性，交互采用“预设选项为主、自由输入为辅”的策略：

- 主演示路径使用预设选项，确保任务推进和情绪映射稳定。
- 自由输入作为增强项存在，在无法稳定识别时回退为中立输入。

### 5.2 设计关键点

玩家输入不直接决定 NPC 文本。玩家输入首先影响 NPC 的内部状态，随后由内部状态决定行为，再由行为生成文本。这样才能体现“情感计算驱动叙事”的系统设计逻辑。

## 6. 核心算法设计

### 6.1 算法名称

情感感知 - 记忆检索 - 任务约束融合的 NPC 叙事决策算法

### 6.2 输入与输出

输入定义为：

- `x_t`：玩家当前输入
- `c_t`：当前场景上下文
- `s_t`：NPC 当前状态
- `M`：NPC 历史记忆集合
- `G`：任务目标与当前阶段

其中 `s_t` 至少包含：

- 当前情绪状态
- 好感度
- 信任度
- 警惕值
- 是否掌握关键线索
- 当前剧情阶段

输出定义为：

- `a*`：NPC 当前最优行为
- `y_t`：NPC 回复文本
- `s_(t+1)`：更新后的 NPC 与任务状态

### 6.3 算法流程

算法分为四个阶段：

#### 阶段一：情感感知

对玩家输入执行简化情绪分类。为控制工期，本项目只做三分类：

- 友好
- 中立
- 冒犯

输出既可以是单标签，也可以是简单分值向量，用于参与后续行为评分。

#### 阶段二：记忆检索

从当前 NPC 的历史记忆中检索相关记忆。第一版采用轻量规则而非向量数据库，综合以下因素评分：

- 关键词匹配程度
- 标签匹配程度
- 记忆重要度
- 新近度

最终取 Top-K 记忆参与本轮决策。

#### 阶段三：任务约束过滤

根据任务状态、角色身份和剧情阶段过滤非法行为。例如：

- 守卫在未建立信任或任务未推进前不能放行
- 村长在主线未推进前不能透露终局线索
- 商人不能发布不属于自身职责的任务

该阶段的目标是先保证行为合法，再在合法集合中选择最优解。

#### 阶段四：候选行为评分

为 NPC 预设少量可控候选行为，例如：

- 安抚
- 回避
- 试探
- 提示线索
- 发布任务
- 拒绝
- 降低信任

对每个候选行为执行评分：

`Score(a) = TaskFit + EmotionFit + MemoryFit + RelationFit - ConstraintPenalty`

取最高分行为作为最终结果，再进入文本生成。

### 6.4 文本生成策略

文本生成采用两层结构：

- 第一层：模板生成，保证系统稳定和可控
- 第二层：可选 LLM 润色，提高自然语言表现

项目答辩时需要明确说明：系统的核心创新是“选行为”，不是“让大模型自由输出文本”。

## 7. 场景、角色与任务设计

### 7.1 场景

- 村庄：主交互场景，承载主要剧情与任务流转
- 森林入口或废塔外部：任务推进场景，用于线索验证和剧情过渡

### 7.2 NPC

- 村长：主线引导者，负责关键剧情信息控制
- 守卫：约束型 NPC，用于体现放行条件和信任门槛
- 商人：支线型 NPC，用于体现关系变化、任务奖励和线索交换

### 7.3 任务结构

- 主线任务：调查村外异常事件并获取关键线索
- 支线任务一：帮助商人找回遗失物，提升关系并获得额外线索
- 支线任务二：满足守卫放行条件，体现任务约束与信任变化

## 9. 接口设计

系统保留 6 个核心接口：

### 9.1 `POST /api/dialogue/interact`

请求参数：

- `player_id`
- `npc_id`
- `scene_id`
- `input_text`
- `selected_option`

返回内容：

- `npc_reply`
- `emotion_result`
- `chosen_action`
- `relationship_change`
- `quest_update`

### 9.2 `GET /api/npc/{npc_id}/state`

返回当前 NPC 的情绪、关系值、最近记忆摘要和任务相关状态。

### 9.3 `GET /api/quest/list`

返回玩家当前任务列表与阶段。

### 9.4 `POST /api/quest/update`

用于推进任务进度。即使后续部分更新逻辑集成在 `interact` 内部，也保留该接口以保证边界清晰。

### 9.5 `POST /api/save/create`

用于保存当前状态。

### 9.6 `GET /api/save/load`

用于读取存档并恢复状态。

接口边界原则是：前端只调用接口，不直接改数据库。所有任务推进、关系变化、对话记录统一在后端完成。

## 10. 异常处理

系统必须为以下异常准备降级路径：

- 后端不可用：前端提示稍后重试，不能卡死
- NPC 无合法行为：回退到默认行为，如回避或保守回复
- 玩家输入无法识别：默认归类为中立输入
- 任务状态异常：统一返回阻断状态，由前端提示条件不足
- 存档恢复失败：回退到最近一次有效状态

系统允许“不够智能”，但不允许“直接崩溃”。

## 11. 测试与验证方案

### 11.1 功能测试

验证以下基础流程是否可用：

- 玩家移动
- NPC 交互触发
- 对话 UI 展示
- 任务接取、更新、完成
- 存档与读档
- 场景与状态刷新

### 11.2 算法测试

准备固定案例，记录：

- 玩家输入
- 情绪识别结果
- 检索到的记忆
- 最终行为
- 输出文本

建议至少覆盖以下情境：

- 玩家友好输入，村长倾向于提供提示
- 玩家冒犯守卫，守卫拒绝放行
- 完成商人支线后，商人态度改善并提供额外线索

### 11.3 用户体验测试

邀请 3 至 5 名同学试玩，评价以下内容：

- NPC 回复是否合理
- 任务推进是否清晰
- 玩家选择是否有实际影响
- 整体互动体验是否符合 RPG 预期

## 12. 一个月开发排期

### 第 1 周：基础骨架

完成：

- Godot 项目和基础场景
- FastAPI 基础项目
- SQLite 接入
- Godot 与后端最小通信
- 1 个 NPC 的最小对话 Demo

阶段目标：

玩家可以在场景中移动并成功触发一次对话。

### 第 2 周：系统闭环

完成：

- 任务表和任务推进
- 关系值变化
- 对话日志记录
- 基础存档
- 三个 NPC 的基础配置

阶段目标：

主线和至少 1 条支线能完整跑通。

### 第 3 周：核心算法

完成：

- 情绪识别模块
- 记忆检索模块
- 任务约束过滤
- 候选行为评分
- 模板文本生成
- 可选 LLM 润色接口预留

阶段目标：

NPC 回复明显受情绪、记忆和任务状态影响。

### 第 4 周：收尾与答辩材料

完成：

- 缺陷修复与参数调整
- UI 细节补齐
- 测试案例整理
- 论文系统设计与算法设计部分整理
- 答辩 PPT 与演示流程准备

阶段目标：

系统可以稳定演示，论文核心章节具备可写内容。

## 13. 最终结论

本项目应以“内容少而完整、算法先于润色”为总原则推进。最终产出不是一个大体量 RPG，而是一个围绕 NPC 智能交互构建的可演示 RPG 原型。该方案兼顾系统完成度与论文性，适合在一个月工期内完成毕业设计与答辩准备。
